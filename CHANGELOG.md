# Changelog

All notable changes to this project will be documented in this file. The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- **Support wildcard `*.domain` entries in `allowedNetworkList`.** An exact-match-only allowlist could not express "this domain and the CDN it redirects to", which silently broke every image served by a host that 302s to a CDN subdomain. `picsum.photos`, for example, redirects to `fastly.picsum.photos`; because the SSRF redirect loop re-validates every hop against the allowlist (correctly), the CDN hop was rejected and each request fell back to a placeholder. An entry may now take the form `*.example.com`, which matches the apex (`example.com`) **and** any subdomain (`cdn.example.com`, `a.b.example.com`). Matching uses a leading-dot suffix test, so a sibling-label host (`evilexample.com`) and a mid-string host (`example.com.evil.net`) are both rejected — there is no suffix-confusion bypass. A wildcard must carry at least two non-empty labels after `*.`, so an overly broad `*.com` (or `*.com.`/`*..com`, which an empty label would otherwise smuggle past the count) is rejected at `registerServe()`; public-suffix families such as `*.co.uk` are deliberately not special-cased, and an operator listing one accepts every host beneath it. **Security note:** a wildcard relaxes only the hostname allowlist. Every redirect hop is still independently re-validated against the http/https protocol guard and the DNS public-IP check, so a wildcard can never open an SSRF path to a private, loopback, or link-local address. Exact entries are matched exactly, as before, so existing configurations are unaffected. (`src/schema.ts`, `src/functions.ts`, `src/types.ts`)
- **Export `resolvePinnedAddresses(hostname)`.** Returns **every** validated public `{ address, family }` pair for a hostname (or `null` if the host is empty, resolves to nothing, resolves to — or is — a private address, or DNS fails), so a caller can pin the full validated set into an agent's `lookup` and still let Happy-Eyeballs fail over across them. The pre-existing single-address `resolvePinnedAddress` is retained unchanged as a thin wrapper returning the first validated pair. (`src/functions.ts`, `src/index.ts`)

### Changed

- **Widen the request dimension window from `[50, 4000]` to `[1, 4000]`.** The framework hard-rejected any request `width`/`height` below 50, turning ordinary small-image requests — 32px and 48px avatars, 16px favicons, dense-list thumbnails — into fallback placeholders, and `optionsSchema` additionally refused to start when `minWidth`/`minHeight` were configured below 50, so an operator could not opt out. The floor is now 1: a request inside `[1, 4000]` validates and is then clamped to the operator's `minWidth`/`maxWidth`/`minHeight`/`maxHeight` as before, and operator minimums may be set anywhere from 1 upward. The obsolete lower-bound refinements are removed (`.positive()` already guarantees `>= 1`); the `maxWidth`/`maxHeight` `<= 4000` ceiling and the `min <= max` refinements are unchanged, as is the upper-bound rejection (a width above 4000 still returns a fallback rather than being clamped down). Defaults are unchanged at `minWidth`/`minHeight` `50` and `maxWidth`/`maxHeight` `4000`. **Behavior-change note (SemVer honesty):** this only loosens validation — nothing previously accepted changes behavior — but a sub-50 request that previously returned a placeholder now returns a genuine image clamped up to the operator's configured minimum. Operators who want true sub-50 output must lower `minWidth`/`minHeight` accordingly. (`src/schema.ts`)

### Fixed

- **Stop the DNS-pinned agents from disabling Happy-Eyeballs, which could fail every outbound fetch on a dual-stack host.** `resolvePinnedAddress` resolved a hostname, validated the answers, then pinned only `addresses[0]` into the agent's `lookup`, while `buildPinnedAgents` additionally set `autoSelectFamily: false` and a single pinned `family`. Because the resolver order is preserved verbatim, a host whose first returned address is an AAAA record would be connected to over IPv6 only — with Happy-Eyeballs explicitly disabled there was no fallback to its perfectly reachable IPv4 address, so on any machine with broken or absent IPv6 egress **every** network image fetch stalled until `requestTimeoutMs` and then served a placeholder. The redirect loop now resolves via `resolvePinnedAddresses` and pins the **full** validated address set; `buildPinnedAgents` sets only `lookup`, leaving Node's default `autoSelectFamily` in place so the connect attempt can fail over across families. This does not weaken the SSRF boundary: the socket's only source of addresses remains the pinned `lookup` closure, every address in it was validated public before the connect, the "reject the whole host if **any** resolved address is private" stance is unchanged, and the kernel resolver is still never consulted again — so the DNS-rebinding TOCTOU window stays closed. `buildPinnedAgents` accepts the new `PinnedAddress[]` form in addition to its original `(address, family)` signature, so external callers are unaffected. (`src/functions.ts`)

## [2.11.0] - 2026-07-03

### Fixed

- **Stop caching a soft fallback (missing/oversized/invalid local file, blocked host, SSRF reject, transient network failure) under the real-image cache policy.** A soft fallback is served through the same happy `200` path as a genuinely resolved image, but it inherited that path's full `Cache-Control` (`public, max-age=86400, stale-while-revalidate=604800` by default) and, for an external `src`, the stable pre-fetch `url:<src>` deterministic ETag — computed before the fetch was even attempted. A transient failure (a blocked host, a momentarily-down origin, a missing file later restored) was therefore cached as if it were the real, permanent image: a client or downstream CDN that cached the placeholder under that ETag could stay pinned to it for a full day, and a client presenting that ETag after the source recovered could get `304 Not Modified`d back onto the placeholder forever. Two new module constants, `DEFAULT_CACHE_CONTROL` (the previous default, unchanged) and `FALLBACK_CACHE_CONTROL` (`public, max-age=60`, matching the value the hard-fallback catch already used), are now the single source of truth for both cache-control strings. `serveImage` clears any deterministic ETag the instant `resolveBuffer()` resolves to a soft fallback, so the response falls through to the buffer-hash ETag block and is keyed on the actual placeholder bytes sent, and the `200` response `Cache-Control` is `FALLBACK_CACHE_CONTROL` whenever a soft fallback was served. **Behavior change (SemVer-honesty note):** an operator who relied on placeholder images being cached for a full day will now see them cached for 60 seconds instead — that is the entire point of the fix. Genuinely resolved images are unaffected: their `Cache-Control` and ETag are unchanged. (`src/pixel.ts`, `src/pixel.test.ts`)
- **Echo `ETag`, `Cache-Control`, and `Vary: Accept-Encoding` on `304 Not Modified` responses (RFC 9110 §15.4.5).** Both `304` short-circuits (the pre-Sharp deterministic-ETag match and the post-Sharp buffer-hash-ETag match) called `res.status(304).end()` with no headers set beforehand — Express's `res.end()` performs no automatic header handling, so a `304` carried none of the validators a client or intermediate cache needs to confirm freshness. Both sites now set `ETag`, `Vary: Accept-Encoding`, and a `Cache-Control` that mirrors what the corresponding `200` would have sent: the pre-Sharp site (which can only ever match a genuine deterministic ETag, never a placeholder's) always uses `cacheControl ?? DEFAULT_CACHE_CONTROL`; the post-Sharp site uses the same fallback-aware value the `200` path uses, so a repeat `304` for a still-missing source correctly reports `max-age=60` instead of the long-lived default. (`src/pixel.ts`, `src/pixel.test.ts`)

### Security

- **Add `X-Content-Type-Options: nosniff` to every image response.** Every path that sends a response body — the `200` happy path, the `200` soft-fallback path, and the `200` hard-fallback (outer-catch) path — now sets `X-Content-Type-Options: nosniff`, stopping a browser from MIME-sniffing the body away from the declared `Content-Type`. The bodyless `304` short-circuits don't need it. (`src/pixel.ts`, `src/pixel.test.ts`)

## [2.10.0] - 2026-07-03

### Added

- **Add a `fallback` field to the `onComplete` completion context.** `PixelServeCompletionContext` gains `fallback: boolean` — `true` when the served bytes are a bundled placeholder image (either a "soft" fallback, e.g. a missing file or blocked host that still gets re-encoded through Sharp on a 200, or a "hard" fallback, the outer pipeline catch serving the bundled asset verbatim), `false` for a genuinely resolved-and-encoded image. A 304 short-circuit always reports `false` — no bytes are sent that round-trip, so there is nothing to characterize as fallback-or-not, even when the underlying resource is itself a fallback. The mark is threaded via a new optional trailing `onFallback?: () => void` callback parameter on `readLocalImage`, and a new optional `onFallback` field on the existing options object accepted by `fetchImage` and by the module-private `fetchFromNetwork` — all additive and backward-compatible, fired whenever the call resolves to `FALLBACKIMAGES[type]()`. `serveImage` instantiates a fresh per-request closure for this callback on every invocation (never module- or factory-scoped), so concurrent requests cannot leak the mark between each other. Additive, non-breaking change to `PixelServeCompletionContext` and to the exported `readLocalImage`/`fetchImage` signatures. (`src/types.ts`, `src/functions.ts`, `src/pixel.ts`)

### Fixed

- **Fire `onComplete` on the hard-fallback path so every 200 fires the hook exactly once.** When the pipeline threw at any point the outer `catch` in `serveImage` could reach — whether before source resolution (e.g. a schema/validation error) or after (e.g. a Sharp decode/encode failure or an SVG rejection) — the catch served the bundled placeholder as a 200 response but never invoked `onComplete` at all — a consumer relying on the hook for latency/volume metrics would silently undercount these responses, with no signal to distinguish "the hook didn't fire because nothing happened" from "the hook didn't fire because this response was a hard fallback." The outer catch now calls `safeOnComplete` after `res.send(fallback)` with `fallback: true`, `cached: false`, `outputBytes` set to the actual served placeholder's length, and `format` derived from the same `fallbackType`-based logic the response headers already use (`png` for avatar, `jpeg` for normal). The 200 happy path and both 304 short-circuits are unaffected in shape — they now also explicitly set `fallback` (`servedSoftFallback` on the happy path, always `false` on a 304), completing the field added above. (`src/pixel.ts`)
- **Make the `defaultQuality` option actually take effect.** `userDataSchema`'s `quality` field carried its own hard-coded `.default(80)`, so a request that omitted `quality` always resolved to a schema-level `80` before `renderUserData` ever ran — its `parsed.quality ?? bounds.defaultQuality` fallback could never fire, making the documented `defaultQuality` option (`registerServe({ defaultQuality: 60 })`) dead code for every caller who didn't also pass an explicit per-request `quality`. The `quality` pipe now ends in `.optional()` instead of `.default(80)`, so an omitted request quality reaches `renderUserData` as `undefined` and its existing `?? bounds.defaultQuality` fallback finally governs. `optionsSchema.defaultQuality` keeps its own `.default(80)`, so the effective middleware default for a `registerServe()` call with no `defaultQuality` override is unchanged at `80`. **Behavior-change note:** a consumer who parses the exported `userDataSchema` directly (bypassing `renderUserData`) and relied on an omitted `quality` being filled to `80` will now see `undefined` — this raw-schema contract change is the necessary price of making the documented option work, and is called out here per SemVer honesty even though the middleware's own effective behavior is unchanged. (`src/schema.ts`)
- **Key the deterministic ETag for an internal-host URL `src` on the underlying local file instead of the immutable URL string, and degrade to the buffer-hash ETag for an oversized local file.** Two related gaps in `buildSourceIdentifier`: (1) a request whose `src` is an `http(s)` URL pointing back at the app's own configured `websiteURL` (e.g. `?src=http://localhost/api/v1/logo.png`) is served from local disk by `fetchImage`, but the ETag was keyed on the literal `url:<src>` string — since that string never changes, overwriting `logo.png` on disk never changed its ETag, so a client that had cached the old image kept receiving `304 Not Modified` for it forever; (2) a local file larger than the configured `maxDownloadBytes` is served as the fallback placeholder image by `readLocalImage`, but the ETag was still keyed on the real (oversized) file's `mtime:size` — decoupled from the fallback bytes actually sent, so raising `maxDownloadBytes` later would incorrectly `304` a client that had cached the placeholder instead of serving the now-eligible real image. `buildSourceIdentifier` now accepts an optional third argument `{ websiteURL, apiRegex, apiPrefix, maxBytes }`: an internal-host URL `src` is resolved to its on-disk pathname via a new shared `resolveInternalLocalPath` helper (extracted from, and now also used by, `fetchImage`'s own internal-host detection, so the two rules cannot drift apart) and falls through to the existing local-file `mtimeMs:size` branch; that branch now returns `null` — degrading to the post-Sharp buffer-hash ETag — when the file's real size exceeds `maxBytes`, mirroring `readLocalImage`'s own size guard. `registerServe`'s call site threads `websiteURL`/`apiRegex`/`apiPrefix`/`maxDownloadBytes` through so the computed identifier always matches the branch that will actually serve the request. The new parameter is optional and additive — existing callers passing only `(src, baseDir)` see no behavior change. (`src/pixel.ts`, `src/functions.ts`, `src/index.ts`)

### Security

- **Extend `isPrivateIp`'s SSRF blocklist to cover RFC 6598 CGNAT space and 6to4 tunneling.** Three gaps: (1) `100.64.0.0/10` (RFC 6598 shared address space — the range cloud providers commonly assign to CGNAT gateways and pod/node-internal networking, e.g. AWS EKS) was not recognized as private, so a fetch target inside that range passed the SSRF guard unblocked; (2) the IPv4 6to4 anycast relay `192.88.99.0/24` (RFC 3068, deprecated) was likewise unblocked; (3) the IPv6 6to4 prefix `2002::/16` (RFC 3056) was not unwrapped at all, so a 6to4 literal embedding a private IPv4 — e.g. `2002:a00:1::` (embedding `10.0.0.1`) or `2002:7f00:1::` (embedding `127.0.0.1`) — passed straight through unblocked. `100.64.0.0/10` and `192.88.99.0/24` are now blocked outright in the IPv4 branch — no legitimate public image host lives in either range. `2002::/16` is handled like the existing NAT64 `/96` prefix: the embedded 32-bit IPv4 address (carried in hextets 1 and 2, per RFC 3056's `2002:<hi>:<lo>::/48` layout) is extracted and re-validated through `isPrivateIp` recursively, so a 6to4-wrapped private/loopback address is blocked while a 6to4-wrapped public address (e.g. `2002:808:808::`, embedding `8.8.8.8` — 6to4 legitimately tunnels arbitrary public IPv4 traffic) still passes. The IPv4-mapped path (`::ffff:100.64.0.13`) already recurses into the IPv4 branch via the existing `::ffff:` handling, so no separate IPv6 case was needed for the CGNAT or 6to4-relay ranges. (`src/functions.ts`)
- **Disable axios' ambient `HTTP(S)_PROXY` env-var proxy detection on the outbound image fetch.** `requestNoRedirect`'s `axios.get` call omitted `proxy: false`, so a proxy configured via the process environment (`http_proxy`/`https_proxy`/`HTTP_PROXY`/`HTTPS_PROXY`) was silently honored by axios. That reopens exactly the window the allowlist + public-IP validation + DNS-pinned `lookup` exist to close: an IP-literal proxy would connect the socket to a target that was never checked against `isPrivateIp`/`allowedNetworkList`, and a hostname-literal proxy re-resolves at connect time through the proxy's own resolver, bypassing the pinned address entirely (the classic DNS-rebinding TOCTOU the pinning was added to prevent). `requestNoRedirect` now sets `proxy: false` unconditionally — axios' own documented mitigation for this exact scenario (its THREATMODEL.md lists "Proxy environment variable hijack," with `config.proxy: false` as the fix). This is the only `axios.get` call in the package, so a single edit closes the gap for every outbound fetch, including every hop of the manual redirect loop. Operators who need an egress proxy must front the middleware with one at the network layer instead. (`src/functions.ts`)
- **Detect `<!DOCTYPE …>`-prefixed SVG — including billion-laughs entity bombs — in the `looksLikeSvg` magic-byte sniffer.** The latin1/UTF-8 branch only ran its `<svg[\s>]` window scan for a head starting with `<?xml` or `<!--`; a buffer starting directly with `<!DOCTYPE svg …>` (a common, spec-legal way to open an SVG document, with no XML declaration) fell through to an unconditional `return false`, so `allowSvgInput:false` never rejected it and it reached Sharp's own parser instead. Worse, even a DOCTYPE-prefixed buffer that WAS scanned could still evade detection: a hostile DTD internal subset (`<!DOCTYPE svg [ <!ENTITY … > … ]>`) padded past the sniffer's 4 KiB inspection window pushes the literal `<svg` root tag out of view, so the window scan alone finds nothing. `<!doctype` is now recognized alongside `<?xml`/`<!--` as a prolog that triggers the window scan (catching the plain in-window case), and independently, a `<!DOCTYPE svg` declaration naming `svg` as the root element (optionally followed by whitespace, `[`, or `>`) is treated as conclusive on its own, regardless of where — or whether — the `<svg` root tag itself falls within the scanned window, since a DOCTYPE naming `svg` as its root element is an unambiguous signal that does not depend on finding the tag. This DOCTYPE-root check is shared across every prolog branch — it fires whether the hostile DOCTYPE opens the buffer directly OR sits behind an `<?xml …?>` declaration or `<!-- … -->` comment (the most common real-world SVG opening is `<?xml version="1.0"?>` then `<!DOCTYPE svg …>`), in both the latin1/UTF-8 and UTF-16 BOM branches — so a bomb DTD cannot slip through by hiding behind an XML declaration or comment prolog. This is a targeted, prolog-specific addition, not a blanket `/<svg[\s>]/` scan of arbitrary binary: the DOCTYPE-root check runs only on a buffer that already begins with a recognized XML prolog, so a raster image whose metadata happens to contain the bytes `<svg` (or `<!doctype svg`) is never inspected and cannot be false-positived. Confirmed empirically that this closes a real gap, not merely a theoretical one: `sharp.prototype.metadata` is provably never invoked on a DOCTYPE-entity-bomb-shaped buffer after the fix (it was invoked once before), meaning the buffer no longer reaches Sharp/librsvg's DTD parser at all — response-shape assertions alone (status/content-type) could not distinguish before from after, since Sharp's own independent `meta.format === "svg"` guard already produced an identical-looking rejection downstream, but only after its parser had already processed the hostile DTD far enough to identify the format. Bringing the UTF-16 branch into structural parity with the latin1/UTF-8 branch also removed a pre-existing UTF-16-only false positive: that branch previously ended in an **ungated** `<svg[\s>]` scan (with no prolog requirement), so a UTF-16 BOM-prefixed plain-text buffer that merely contained the substring `<svg` was wrongly classified as SVG and over-blocked; the scan is now gated on a recognized XML prolog in both branches, matching the latin1 path's existing behavior. (`src/pixel.ts`)

## [2.9.0] - 2026-07-02

### Fixed

- **Fix DNS-pinned agent `lookup` breaking every real network image fetch on Node >=20.** `buildPinnedLookup` only implemented the legacy 3-argument callback shape (`callback(null, address, family)`). Since Node 20, `net.getDefaultAutoSelectFamily()` defaults to `true`, and when no `family` is pinned on the connect options, Node invokes an `Agent`'s pinned `lookup` with a `{ all: true }` options object and expects `callback(null, [{ address, family }])` — an array. The single-address form resolved to `undefined` under that calling convention, so `net.Socket.connect()` threw `TypeError [ERR_INVALID_IP_ADDRESS]: Invalid IP address: undefined`; `requestNoRedirect` swallowed the error and every network fetch silently fell through to the placeholder image instead of the real one. Because the existing test suite mocks `axios.get` directly, none of those tests ever reached the real `net.Socket.connect()` path, so the regression shipped invisibly. `buildPinnedLookup` now branches on `options.all` and answers with the array shape when it is set, the legacy single-address shape otherwise — matching Node's own `net.LookupFunction` type exactly, so the fix is enforced by `tsc --noEmit` rather than a runtime cast. Reproduced end-to-end on Node v24.14.0 with a real (un-mocked) request against a loopback server before the fix, and confirmed resolved after. (`src/functions.ts`)
- **Normalize `websiteURL` so `host:port` and full-URL config forms are recognized as internal.** `fetchImage` compared `url.hostname` against the raw `websiteURL` string. A bare hostname (`"example.com"`) happened to match by coincidence, but the `host:port` form (`"example.com:8080"` — the shipped demo config) and a full URL (`"https://example.com:8080"`) never did, because `url.hostname` never carries a port and never carries a scheme: `new URL("example.com:8080")` (no scheme prepended) parses as protocol `"example.com:"` with opaque path `"8080"`, not an authority with a host. Internal images configured either of those two ways were silently misrouted through the external-fetch/`allowedNetworkList` path instead of being read locally. Added a `normalizeWebsiteHost` helper that parses `websiteURL` as a URL (prepending a placeholder `http://` scheme when the configured value has none) and derives both `hostname` (no port) and `host` (with port); `fetchImage` now treats a request as internal when the request URL's `hostname` OR `host` matches the normalized value or its `www.` variant. A configured value that fails to parse even with the placeholder scheme falls back to the previous raw-string comparison, so no previously-working configuration can regress. The `optionsSchema`-validated, stored `websiteURL` value is unchanged — this is purely a comparison-time fix. (`src/functions.ts`)
- **Reject misconfigured `minWidth`/`maxWidth`/`minHeight`/`maxHeight` at `registerServe()` instead of shipping a silently non-functional bound.** `userDataSchema` hard-rejects any request `width`/`height` outside `[50, 4000]` before `renderUserData`'s `clamp()` step ever runs, so an operator-configured bound outside that window could never actually take effect for the out-of-window portion of its range — e.g. `minWidth: 10` could never rescue a `width=20` request, since the schema always threw `"width too small"` first and the request silently fell back to the placeholder image instead of the documented "clamped to bounds" behavior. `optionsSchema` now adds two more refinements — `minWidth >= 50 && maxWidth <= 4000` and `minHeight >= 50 && maxHeight <= 4000` — so a misconfigured bound throws a clear, actionable error at startup instead of shipping quietly broken. The `userDataSchema` per-request throw-on-out-of-range behavior (and its fallback-image contract) is unchanged — this is a purely additive guardrail at the options layer. (`src/schema.ts`)
- **Fix the outer-catch fallback mislabeling the PNG avatar image as `image/jpeg`.** When the pipeline threw *after* the source buffer was resolved — e.g. a Sharp decode/encode failure, or an SVG rejection on an `avatar`-type request — the outer `catch` in `serveImage` served the pre-encoded bundled fallback asset verbatim (deliberately skipping a Sharp re-encode) but hardcoded `Content-Type: image/jpeg` and `Content-Disposition: filename="fallback.jpeg"` regardless of the requested type. For an `avatar` request that fallback is `noavatar.png` (a genuine PNG), so the response bytes and the declared MIME type disagreed. The catch now derives the format from `fallbackType` (`png` for avatar, `jpeg` for normal) and sets the `Content-Type`, filename extension, and served bytes consistently. The 200/304 success paths were already correct — the missing-file fallback there is re-encoded through Sharp to the requested output format — so only this defensive error path was affected. (`src/pixel.ts`)

### Security

- **Pin `family` and disable `autoSelectFamily` on both pinned agents (defense in depth).** `buildPinnedAgents` now also passes the already-resolved `family` and `autoSelectFamily: false` directly to both the `http.Agent` and `https.Agent` it builds, on top of the array-aware pinned `lookup` above. This keeps the DNS-rebinding mitigation (the pinned-`lookup` approach documented in the README's Security Features section) robust even if a future Node release changes how or when an `Agent`'s `lookup` option is consulted. (`src/functions.ts`)
- **Gate `buildSourceIdentifier`'s local-file `fs.stat` behind `isValidPath` (ETag/cache-key oracle).** The local-file branch resolved `path.resolve(baseDir, src)` and called `fs.stat` on it directly, with no containment check. A traversal `src` (e.g. `"../secret.txt"`) that happened to reference an existing file outside `baseDir` still `fs.stat`ed successfully, producing a deterministic `file:<mtimeMs>:<size>` identifier baked into the response `ETag` — leaking that out-of-tree file's mtime/size as an oracle, and decoupling the `ETag` from the bytes actually served (the pipeline's own `readLocalImage` already rejects the same traversal internally via its own `isValidPath` check and serves a fallback image, so the response body and its `ETag` no longer corresponded to the same source). `buildSourceIdentifier` now calls `isValidPath(baseDir, src)` before ever touching `fs.stat`; when it returns `false` the source identifier degrades to `null` and the pipeline falls back to the post-Sharp buffer-hash `ETag`, which is always derived from the bytes actually sent to the client. (`src/pixel.ts`)
- **Harden `isPrivateIp` against deprecated IPv6 site-local and NAT64-wrapped private addresses.** Two gaps: (1) the deprecated site-local range `fec0::/10` (RFC 3879) was not recognized as private; (2) NAT64-translated addresses (RFC 6052) were not unwrapped, so an SSRF target that resolved to (or was supplied as) a NAT64-embedded private IPv4 — e.g. `64:ff9b::7f00:1` (`127.0.0.1`) or `64:ff9b::a00:1` (`10.0.0.1`) — passed the private-IP check unblocked. Both are now covered: `fec0::/10` is flagged outright, and any address under the well-known prefix `64:ff9b::/96` has its embedded IPv4 extracted (handling both the hextet tail, e.g. `:7f00:1`, and the dotted-quad tail, e.g. `:127.0.0.1`) and re-validated through `isPrivateIp` recursively, so a wrapped private address is blocked while a wrapped public address still passes through unaffected. The NAT64 prefix is matched by fully expanding the address into its 8 numeric hextets first (a new internal `expandIPv6Hextets` helper, handling `::` compression at any position) rather than a `startsWith("64:ff9b::")` string check, so a fully-expanded or partially-compressed representation of the exact same address cannot bypass the match. The local-use NAT64 prefix `64:ff9b:1::/48` (RFC 8215) is deliberately **not** unwrapped — RFC 8215 explicitly exempts it from the well-known prefix's "embedded address must be public" restriction (it may legitimately carry private addresses) and it is scoped only to an operator's own limited/local NAT64 domain, so the entire range is treated as private without attempting the non-contiguous RFC 6052 §2.2 bit-extraction a `/48` prefix would require (two independent judge reviews converged on this over a precise-extraction alternative, given it would be new, untested, security-critical bit-splicing logic in a function where a subtly wrong decode is a strictly worse failure mode than over-blocking an obscure, non-globally-reachable range). (`src/functions.ts`)
- **Classify every IPv6 address numerically in `isPrivateIp` — fix multiple SSRF bypasses from textual (non-numeric) matching.** The remaining IPv6 checks (`::`, `::1`, `::ffff:…`, `fe80::/10`, `fec0::/10`, `fc00::/7`, `ff00::/8`) still matched on the address's *text* — an exact-string `=== "::1"`, a `startsWith("::ffff:")`, and leading-hextet regexes. Because a single IPv6 address has many equivalent textual spellings, this was unsound in several ways, each confirmed against the exported guards (`isPrivateIp`, and through them `isPublicHost` / `resolvePinnedAddress`, which callers are told to use on their own outbound requests and which also back the middleware's own DNS-validation path):
  - the loopback `::1` written uncompressed (`0:0:0:0:0:0:0:1`), the unspecified `::` written in full, and an IPv4-mapped address written uncompressed (`0:0:0:0:0:ffff:7f00:1` = `::ffff:127.0.0.1`) all slipped past the checks and were reported **public**;
  - the deprecated IPv4-compatible form `::a.b.c.d` (RFC 4291 §2.5.5.1, distinct from the `::ffff:` mapped form) embedding a private/loopback IPv4 — e.g. `::127.0.0.1`, `::10.0.0.1`, `::169.254.169.254` — was reported **public**;
  - an RFC 4007 zone-id suffix on a dotted-quad-tailed literal (`::ffff:127.0.0.1%eth0`, which `net.isIP` accepts as valid IPv6 and the OS will connect to real loopback) was reported **public**, because the hextet expander silently mis-parsed the tail (`parseInt("127.0.0.1%eth0", 16)` → `0x127`) instead of failing closed;
  - conversely `fe8:…` (the hextet `0x0fe8`, in unrelated reserved space, **not** link-local) was wrongly flagged private by the `fe80::/10` regex.

## [2.8.0] - 2026-05-12

### Changed

- **`renderOptions` runs once at factory time, not per request (Task 4).** `registerServe` now validates the `PixelServeOptions` payload through `optionsSchema.parse` exactly **once** when the middleware is constructed and threads the resulting `ParsedOptions` into every `serveImage` invocation. Previously each request re-ran the entire Zod schema (strict-mode unknown-key check, `websiteURL` regex match, `refine()` callbacks for `minWidth <= maxWidth` / `minHeight <= maxHeight`, `allowedNetworkList.transform(trim → lowercase)`, function-type custom validators), allocated a fresh parsed-options object, and re-bound `onError` / `onComplete` to themselves. The hot path is now arithmetic: the schema cost — including any user-supplied regex — is bounded to startup. Operator misconfiguration is surfaced loudly: the schema parse fires `onError` with `phase: "schema"` and re-throws synchronously from `registerServe` so a bad deployment fails fast rather than silently serving fallback images forever. The public API is unchanged — `registerServe(options)` still returns the middleware function — and successful configurations continue to work identically. (`src/pixel.ts`)
- **`buildFilename` no longer truncates mid-percent-encoded byte (Task 5).** The `filename*=UTF-8''...` RFC 5987 parameter value used `encodedBase.slice(0, maxBase)` directly, which could land inside a `%XX` triplet for long unicode source names (e.g., `'中'.repeat(40)` produced trailing `%E4%B`, which is malformed and rejected by strict RFC 3986 / RFC 5987 parsers). The truncation now walks back from the slice boundary to the nearest `%` and drops any incomplete trailing triplet, then walks back past any trailing UTF-8 lead byte that lost its continuation bytes (a complete percent-encoded byte like `%E4` is still invalid UTF-8 on its own — it is the lead byte of a 3-byte sequence). The result decodes cleanly under strict UTF-8 parsers and the overall length cap is still respected. Plain ASCII names are unaffected. (`src/pixel.ts`)
- **`websiteURL` validation regex replaced with a ReDoS-safe pattern (Task 6).** The schema previously used `/^(?![-.])([\w]+[-.]?)*[\w]+$/` whose nested quantifier `([\w]+[-.]?)*` followed by a required `[\w]+$` exhibited catastrophic backtracking on inputs like `"a".repeat(50) + "!"` (a pathological misconfiguration would hang the entire startup before Task 4 — and previously hung every request). The replacement `/^(?!-)[A-Za-z0-9-]{1,63}(\.(?!-)[A-Za-z0-9-]{1,63})*$/` is anchored with no nested repetition: each label is 1-63 alphanumeric/hyphen chars, labels are dot-separated, and no label may start with `-`. Linear-time matching is now guaranteed. Backward compatible for every well-formed hostname — `example.com`, `localhost`, `sub.example.com`, `www.example.com` all still validate, and URLs like `https://example.com/path` continue to flow through the `z.url()` branch of the union. The previous regex's `\w`-driven underscore acceptance is dropped because RFC 1123 hostnames do not include `_`. After Task 4, the regex only runs once at factory time, but the safe pattern still removes the operator-controlled footgun entirely. (`src/schema.ts`)

## [2.6.0] - 2026-05-12

### Security

- **DNS rebinding mitigation via pinned `lookup` (Task 3).** `fetchFromNetwork` previously called `isPublicHost` (which itself runs `dns.lookup`) and then immediately handed the URL to `axios.get`, which performed its OWN kernel-resolver lookup. Between the two lookups, an attacker-controlled authoritative name server could answer the framework's validation with a public IP and respond to axios's lookup microseconds later with `127.0.0.1`/`169.254.169.254` — the canonical DNS-rebinding exploit. The middleware now resolves the hostname once via the new `resolvePinnedAddress` helper, validates every returned address against `isPrivateIp`, then constructs a per-request pair of `http.Agent`/`https.Agent` whose `lookup` function is pinned to that exact `{ address, family }` pair via `buildPinnedAgents`. Axios receives those agents via `httpAgent`/`httpsAgent` so the TCP socket is guaranteed to connect to the IP the framework validated. Each redirect hop in the manual loop independently re-resolves and re-pins, so chained rebinding attempts are also defeated. Backward compatible: no API surface change, no new option. (`src/functions.ts`, `src/functions.test.ts`, `README.md`)

### Changed

- **`looksLikeSvg` hardened (Task 4).** The SVG magic-byte sniffer previously inspected only offset 0 (or offset 3 after a UTF-8 BOM) over a 1 KiB latin1 window, leaving three bypasses: a single 0x20 byte before the UTF-8 BOM (BOM check fired only at offset 0), UTF-16 BE/LE BOM-prefixed SVGs (`toString("latin1")` produced garbage that never matched `<svg`), and pathologically large XML prologs (`<?xml ...?>` plus comments) that pushed `<svg` past the 1 KiB inspection window. The hardened detector now (1) skips leading ASCII whitespace (tab, LF, CR, space) BEFORE the BOM check, (2) recognises UTF-16 BE (`0xFE 0xFF`) and UTF-16 LE (`0xFF 0xFE`) BOMs by re-decoding the head as UTF-16 (byte-swapping for BE) and re-running the heuristic, (3) handles UTF-8 BOM after any leading whitespace, and (4) widens the latin1 inspection window from 1 KiB to 4 KiB so longer XML prologs cannot push `<svg` out of view. Sharp's `meta.format === "svg"` post-decode check remains in place as belt-and-suspenders. (`src/pixel.ts`, `src/pixel.test.ts`)
- **`res.headersSent` guard in outer fallback catch (Task 5).** The outer `catch` in `serveImage` previously called `res.type / res.setHeader / res.send` unconditionally, assuming no bytes had been flushed yet. ## [2.0.0] - 2026-05-12

### Breaking Changes

- **SVG removed from supported output formats.** Sharp/libvips does not support SVG encoding; previously any `format=svg` request silently fell back. The format is now explicitly excluded from the `ImageFormat` union, dropped from `allowedFormats`, and `mimeTypes` no longer maps `svg`. Consumers sending `format=svg` will receive the default `jpeg` output instead. (`src/types.ts`, `src/variables.ts`, `src/pixel.ts`, `src/renders.ts`, `README.md`)

### Security

- **SSRF mitigation for HTTP redirects (Critical).** `fetchFromNetwork` previously let axios auto-follow up to five redirects with no per-hop validation. An allowed host could redirect to private/loopback/link-local addresses such as the AWS IMDS endpoint (`169.254.169.254`), internal Redis (`127.0.0.1:6379`), or any RFC1918 range. The implementation now sets `maxRedirects: 0` on axios and runs a manual redirect loop (default budget: 3 hops, capped at 10). Every hop re-checks the protocol, the `allowedNetworkList`, and a DNS-based public-IP guard. (`src/functions.ts`)
- **Public-host helpers.** Added `isPrivateIp(ip)` and `isPublicHost(hostname)`. `isPrivateIp` rejects RFC1918, loopback, link-local (incl. IPv6 `fe80::/10`), unique-local (`fc00::/7`), multicast (`ff00::/8`), `0.0.0.0`, IPv4-mapped private IPv6, and invalid addresses. `isPublicHost` short-circuits for IP literals and otherwise resolves the hostname via `dns.lookup({ all: true })` and rejects when any returned address is private. (`src/functions.ts`)
- **New `maxRedirects` option** in `PixelServeOptions` (default `3`, range `0..10`). Backward compatible. (`src/types.ts`, `src/schema.ts`)
- **Decompression-bomb protection for Sharp (Critical).** Sharp was instantiated with only `failOn: "truncated"` and no `limitInputPixels`. A small encoded payload could decompress to billions of pixels and OOM the worker. Sharp is now constructed with `{ failOn: "warning", limitInputPixels, sequentialRead: true, unlimited: false }`, and the pipeline peeks `image.metadata()` before any heavy decode, bailing when `width * height > maxInputPixels`. (`src/pixel.ts`)
- **SVG input rejection.** Added a magic-byte SVG sniffer (`looksLikeSvg`) that flags buffers starting with `<svg`, `<?xml ... <svg`, a UTF-8 BOM + `<svg`, or `<!-- ... <svg`. When `allowSvgInput` is false (the default), SVG inputs are rejected before reaching libvips/librsvg, preventing XML-bomb / billion-laughs / nested-`<use>` attacks. (`src/pixel.ts`)
- **New `maxInputPixels` option** in `PixelServeOptions` (default `16_000 * 16_000` = 256 megapixels). (`src/types.ts`, `src/schema.ts`)
- **New `allowSvgInput` option** in `PixelServeOptions` (default `false`). (`src/types.ts`, `src/schema.ts`)

### Tests

- Added 16 new SSRF tests in `src/functions.test.ts` covering private IP detection (IPv4 + IPv6, link-local, unique-local, multicast, IPv4-mapped), DNS-based public-host validation, redirect chain length limits, redirects to private IPs, redirects to AWS IMDS, redirects to unlisted hosts, malformed `Location` headers, missing `Location`, protocol-switch redirects, axios error responses with attached `response`, and the `validateStatus` closure. (`src/functions.test.ts`)
- Added Sharp pixel-bomb / SVG-input tests in `src/pixel.test.ts` covering: SVG buffer rejection (magic-byte + Sharp `metadata.format`), `maxInputPixels` short-circuit, `looksLikeSvg` across all input shapes (BOM-prefixed, comment-prefixed, XML-prolog, empty, null), schema-level `format=svg` rejection, and `allowSvgInput=true` opt-in. (`src/pixel.test.ts`)
- Added schema tests for the new options' defaults, overrides, and validation bounds. (`src/schema.test.ts`)
- Updated `src/variables.test.ts` and `src/renders.test.ts` to reflect the SVG removal and to assert it explicitly.

### Documentation

- Documented the new `maxRedirects`, `maxInputPixels`, and `allowSvgInput` options in the `PixelServeOptions` table. (`README.md`)
- Added "SSRF Redirect Protection", "Decompression-Bomb Protection", and "SVG Input Rejection" subsections under "Security Features", describing the manual per-hop revalidation, Sharp pixel-bomb guard, and SVG magic-byte sniffer. (`README.md`)

### Internal

- Tightened `isPrivateIp` IPv6 handling: malformed IPv4-mapped IPv6 literals (e.g. `::ffff:0.0.0`) are now treated as unsafe instead of falling through, and removed the redundant `lower.startsWith("fe8"/...)` fallback chain that duplicated the link-local regex. (`src/functions.ts`)
